# Writing & Distributing Plugins

A plugin is just a library that exposes one or more `ecewo_*` entry points and is linked next to ecewo. There is no runtime plugin discovery: the host registers a plugin by calling its function (e.g. `myplugin_register(app)`) before listening.

Plugins are consumed via `ecewo_add(name@version)`, which fetches the plugin at configure time and builds it together with your app — the model in [11.plugins.md](11.plugins.md). This page covers how to **write** your own plugin.

If you only write application code in C and pull plugins from the registry, you do not need this page.

## The plugin template

A ready-to-copy scaffold lives at [`examples/plugin-template/`](../examples/plugin-template/):

```
examples/plugin-template/
  CMakeLists.txt        # standalone project using FetchContent
  include/hello.h       # public entry point: hello_register(app)
  src/hello.c           # one middleware + one handler + the register fn
```

It builds standalone via FetchContent exactly the way a third-party plugin repository would. Copy the directory, rename `hello` to your plugin, and edit `src/`.

## Writing your own plugin

A plugin exposes an entry point the host calls to register its routes/middleware. The template's looks like this:

```c
/* hello.h */
#include "ecewo.h"

HELLO_EXPORT void hello_register(ecewo_app_t *app);
```

```c
/* hello.c */
#include "hello.h"

static void hello_handler(ecewo_request_t *req, ecewo_response_t *res) {
  ecewo_send_text(res, ECEWO_OK, "Hello from a plugin!");
}

void hello_register(ecewo_app_t *app) {
  ecewo_route_t *r = ecewo_route_new(app, ECEWO_METHOD_GET, "/hello");
  ecewo_route_handler(r, hello_handler);
}
```

Use the route **builder** (`ecewo_route_new` / `ecewo_route_middleware` / `ecewo_route_handler`), not the `ECEWO_GET`/`ECEWO_POST` macros — the macros are C-only and are not usable from a plugin meant for FFI consumers. See [02.defining-routes.md](02.defining-routes.md).

## Building a plugin as a shared library

For a plugin you distribute as a `.so` (e.g. for binding authors to load at runtime), the minimal `CMakeLists.txt` is:

```cmake
cmake_minimum_required(VERSION 3.14)
project(ecewo-hello LANGUAGES C)

if(NOT TARGET ecewo::ecewo)
  include(FetchContent)
  FetchContent_Declare(
    ecewo
    GIT_REPOSITORY https://github.com/ecewo/ecewo.git
    GIT_TAG        v4
    GIT_SHALLOW    TRUE
  )
  FetchContent_MakeAvailable(ecewo)
endif()

add_library(ecewo_hello SHARED src/hello.c)
target_include_directories(ecewo_hello PUBLIC
  $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>)
target_link_libraries(ecewo_hello PRIVATE ecewo::ecewo)
```

Build it:

```sh
cmake -B build
cmake --build build
```

This yields `libecewo_hello.so`. When built alongside ecewo in DIST mode (as binding authors typically do via `set(ECEWO_BUILD_DIST ON)` before `FetchContent_MakeAvailable(ecewo)`), the plugin's `ecewo_*` and `uv_*` references resolve from `libecewo` at load time:

```sh
ldd build/libecewo_hello.so          # -> libecewo.so.4
nm -D --undefined-only build/libecewo_hello.so | grep -E 'ecewo_|uv_'
```

## What a plugin may call

- **The full `ecewo_*` API** in `ecewo.h`.
- **libuv directly.** In a DIST build, `libecewo` re-exports `uv_*`, so a plugin can `#include "uv.h"` and call libuv (e.g. for a raw-socket protocol via `ecewo_connection_takeover`) while linking only against `ecewo::ecewo`.

Do not link a second copy of libuv into a plugin — there must be exactly one libuv in the process, and it is the one inside `libecewo`.

## Using a plugin at runtime

A C host links the plugin and calls its entry point:

```c
#include "ecewo.h"
#include "hello.h"

int main(void) {
  ecewo_app_t *app = ecewo_create();
  hello_register(app);          // plugin registers its routes
  ecewo_listen(app, 3000);
}
```

An FFI/binding host loads both libraries (`libecewo` and the plugin `.so`) and calls the same entry point through its foreign-function interface. Plugin handlers and middleware are ordinary `ecewo_handler_t` / `ecewo_middleware_t` pointers, so the trampoline/closure patterns in [17.ffi-bindings.md](17.ffi-bindings.md) apply unchanged.

## Platform notes

- **Linux / macOS:** the symbol-export filter is applied automatically (a linker version script / exported-symbols list) in DIST builds.
- **Windows:** the core FFI surface works, but a plugin that links `uv_*` from `libecewo` needs the DLL to export those symbols via a `.def`. This is not yet generated for the DIST build; pure-`ecewo_*` plugins are unaffected.

## Versioning

`libecewo` has `SOVERSION` 4. Within the v4.x line the `ecewo_*` ABI is stable. Because plugins resolve `uv_*` from `libecewo` in a DIST build, the bundled **libuv ABI is also part of the distributable's contract** — it is pinned (currently libuv v1.52.0) and only changes on a deliberate bump. Build plugins against the same major version they will run against. See [17.ffi-bindings.md#versioning-and-soversion](17.ffi-bindings.md#versioning-and-soversion).
