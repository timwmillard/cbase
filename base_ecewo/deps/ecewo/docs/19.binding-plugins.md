## Choose the artifact

Bind against the **self-contained distributable**, not the plain shared build. Build it yourself â€” ecewo does not ship prebuilt binaries; bindings build the C side from source as part of their own CI pipeline via FetchContent with `ECEWO_BUILD_DIST=ON`:

```cmake
set(ECEWO_BUILD_DIST ON)
FetchContent_Declare(
  ecewo
  GIT_REPOSITORY https://github.com/ecewo/ecewo.git
  GIT_TAG        v4
  GIT_SHALLOW    TRUE
)
FetchContent_MakeAvailable(ecewo)
```

This gives you a single `libecewo.so` / `.dylib` (libuv and llhttp bundled inside) that exports only `ecewo_*` and `uv_*`. For a binding that is one file to find and `dlopen`, and there is no risk of picking up the wrong `libuv` at load time.

For a worked example of a binding's CI building ecewo from source via CMake `FetchContent` and shipping the result inside platform-specific wheels, see [20.binding-cookbook.md](20.binding-cookbook.md) â€” that's the path the official Python binding ([ecewopy](https://github.com/savashn/ecewopy)) uses, and the recommended pattern for any new binding.

The plain `ECEWO_BUILD_SHARED` build also works, but then `libecewo`, `libuv`, and `libllhttp` are three separate files you must keep together.

## Enumerate the symbol surface

Everything you may call is `ecewo_*` (plus the bundled `uv_*` if you integrate with the loop). Generate the list straight from the artifact:

```sh
# Linux
nm -D --defined-only libecewo.so | awk '{print $3}' | grep '^ecewo_'

# macOS
nm -gU libecewo.dylib            | awk '{print $3}' | grep '^_ecewo_'

# Windows
dumpbin /exports ecewo.dll
```

The signatures for each are in `include/ecewo.h`. Internal symbols (`ecewo__*`, llhttp, rax) are hidden in the DIST build, so this list is exactly what you may wrap.

## The core binding flow

The lifecycle is unchanged from [17.ffi-bindings.md#lifecycle](17.ffi-bindings.md#lifecycle): create an app, register routes via the **builder** (`ecewo_route_new` / `ecewo_route_middleware` / `ecewo_route_handler` â€” never the `ECEWO_GET` macros, which are C-only), then `ecewo_listen`. Handlers are passed as C function pointers, so the trampoline/closure techniques in [16](17.ffi-bindings.md#the-closure-problem) apply.

Pseudocode for a dlopen-style binding:

```text
lib   = dlopen("libecewo.so", RTLD_NOW | RTLD_GLOBAL)   # see note below
app   = lib.ecewo_create()
r     = lib.ecewo_route_new(app, ECEWO_METHOD_GET, "/")
lib.ecewo_route_handler(r, host_trampoline)             # one C-callable per route
lib.ecewo_listen(app, 3000)                             # blocks on the loop
```

## How plugins fit

A plugin is **not** discovered or auto-loaded. It is a separate shared library that exposes its own exported functions, including an entry point you call to register it on an app. The model:

- The plugin `.so` is built alongside ecewo via FetchContent (or as a sibling `add_subdirectory`). It links `ecewo::ecewo`, so its `ecewo_*` and `uv_*` references resolve from `libecewo` â€” the **one** libuv in the process. A plugin must never bundle its own libuv.
- The **official registry plugins support this directly**: each checks `if(NOT TARGET ecewo::ecewo)` before fetching ecewo, so you can pull any of them into a binding's build system (e.g. via `ecewo_add(cookie@v1)`) and they'll link against the same `libecewo` your binding already built. See [18.writing-plugins.md#building-a-plugin-as-a-shared-library](18.writing-plugins.md#building-a-plugin-as-a-shared-library).
- Your binding loads the plugin `.so`, looks up its entry point, and calls it with the `ecewo_app_t *` you created from `libecewo`. Because there is a single `libecewo` in the process, the app pointer is valid across both libraries.

So binding "ecewo + plugins" is: bind core ecewo as above, then for each plugin, load its `.so` and wrap the functions it exports.

## Loading plugins from your binding

Order and visibility matter. A plugin `.so` has undefined `ecewo_*`/`uv_*` symbols that must resolve to the **same** `libecewo` your app lives in.

The robust pattern for dlopen/ctypes-style bindings:

```text
# 1. Load libecewo first, GLOBAL so its symbols satisfy later loads.
ecewo  = dlopen("libecewo.so", RTLD_NOW | RTLD_GLOBAL)

# 2. Now load each plugin. Its ecewo_*/uv_* refs bind to the global libecewo.
plugin = dlopen("libecewo_cookie.so", RTLD_NOW)

# 3. Register the plugin on your app.
plugin.cookie_register(app)
```

Notes:

- **`RTLD_GLOBAL` on `libecewo` is the key.** Without it (default `RTLD_LOCAL`), a plugin loaded separately may fail to resolve `ecewo_*`, or â€” worse â€” pull in a second `libecewo`. One process, one `libecewo`, one loop.
- A plugin built by linking `ecewo::ecewo` lists `libecewo.so.4` in its `NEEDED` entries, so the dynamic loader can also auto-load it; loading `libecewo` yourself first with `RTLD_GLOBAL` just makes the sharing explicit and order-independent.
- Compiled (non-dlopen) bindings that link the plugin and `libecewo` at build time get this for free â€” the linker records the dependency.
- Verify with `ldd libecewo_cookie.so` (should show `libecewo.so.4`) and `nm -D --undefined-only libecewo_cookie.so | grep -E 'ecewo_|uv_'`.

## Wrapping a plugin's API

Each plugin defines its own exported symbols and naming â€” there is no single convention across plugins, so treat each one like its own small library:

1. **Find its public header / docs.** Each plugin repo documents the functions it exports and its entry point (commonly `<plugin>_register(app)` or similar). The [plugin template](./examples/plugin-template/) shows the shape: a `*_register(ecewo_app_t *)` plus whatever helpers the plugin offers.
2. **Enumerate its exports** the same way as core: `nm -D --defined-only libecewo_<name>.so`.
3. **Wrap them like any C API**, reusing the conventions from [16](17.ffi-bindings.md): opaque pointers stay opaque, returned strings are arena-owned (copy them before the handler returns), callbacks are C function pointers needing the trampoline pattern.
4. **Pass ecewo handles through unchanged.** A plugin function that takes `ecewo_request_t *` / `ecewo_response_t *` / `ecewo_app_t *` receives the very pointers your core binding already holds. Per-request state crosses the plugin/handler boundary via `ecewo_context_set` / `ecewo_context_get`.

If a plugin exposes a raw-socket protocol (WebSocket, etc.), it uses `ecewo_connection_takeover` with libuv callbacks â€” your binding produces those callbacks as C function pointers exactly as in [17.ffi-bindings.md#libuv-integration](17.ffi-bindings.md#libuv-integration); the `uv_*` symbols come from the bundled libuv.

## Threading and the shared loop

Nothing about plugins changes the threading model in [17.ffi-bindings.md#threading-model](17.ffi-bindings.md#threading-model): there is one libuv loop, owned by `libecewo`. Plugin handlers, middleware, timer and takeover callbacks all run on that loop thread. A plugin offloading work uses the same `ecewo_spawn` rules. Your host runtime (GIL, BEAM scheduler, etc.) must let ecewo's loop own the calls â€” that rule is process-wide and covers plugin callbacks too.

## Versioning across ecewo and plugins

- `libecewo` carries `SOVERSION` 4; the `ecewo_*` ABI is stable across v4.x. Pin the major you compiled against.
- Because plugins resolve `uv_*` from `libecewo`, the **bundled libuv ABI is part of the contract** (pinned, currently v1.52.0). A plugin and the `libecewo` it loads against must come from compatible builds.
- Each plugin has its **own** version on top of that. Pin both: the ecewo major and the plugin version (the registry's `name@version`). A binding that ships prebuilt `.so`s should ship a matched set built against the same `libecewo`.

## Checklist

- [ ] Built `libecewo` with `ECEWO_BUILD_DIST=ON`; confirmed exports are only `ecewo_*` / `uv_*`.
- [ ] Core binding uses the route **builder**, not the macros; handlers go through a trampoline.
- [ ] `libecewo` is loaded once, `RTLD_GLOBAL`, before any plugin.
- [ ] Each plugin `.so` shows `libecewo.so.4` in `ldd` and has no second libuv.
- [ ] Plugin functions wrapped per the [16](17.ffi-bindings.md) conventions; arena-owned strings copied out.
- [ ] ecewo major + each plugin version pinned to a matched set.
