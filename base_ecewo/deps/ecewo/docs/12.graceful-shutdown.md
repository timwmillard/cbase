## Overview

ecewo handles graceful shutdown automatically. When the process receives `SIGINT` (Ctrl+C) or `SIGTERM` (e.g. `kill`), every registered app:

1. Stops accepting new connections
2. Closes idle keep-alive connections immediately
3. Waits for in-progress requests to finish (up to the configured `shutdown_timeout_ms`, default 15 seconds)
4. Force-closes any connections still open after the timeout

You do not need to set up signal handlers yourself.

In a [multi-app](./15.multi-app.md) process the signal shuts down every app concurrently; each app runs its own shutdown sequence and `ecewo_atexit` callback. `ecewo_run()` returns once all apps have shut down.

## Shutdown Sequence

```
Signal received (SIGINT / SIGTERM), or ecewo_shutdown() called
  │
  ├─ ecewo_is_running() returns false
  ├─ Stop accepting new connections
  ├─ Wait for pending async work (ecewo_spawn() or external functions coming from plugins) to complete
  ├─ Wait for in-progress requests to finish
  ├─ Force-close remaining connections
  │
  ├─ ecewo_atexit callback fires
  │    All requests are done. Event loop is still alive.
  │    Safe to close DB connections, pools, files, and free the resources.
  │
  ├─ Event loop teardown
  └─ Event loop freed
       │
ecewo_run() returns
```

## Resource Cleanup

```c
int ecewo_atexit(ecewo_app_t *app, void (*callback)(void *user_data), void *user_data);
```

Register a callback to run when the server is shutting down. The callback fires after every in-flight request has finished and while the event loop is still alive - making it the single correct place for all cleanup, whether your resources are plain C objects, database connection pools, or plugins that rely on libuv internally.

| Parameter   | Description |
|-------------|-------------|
| `app`       | The application instance |
| `callback`  | Function to call on shutdown |
| `user_data` | Arbitrary pointer passed to the callback. Use it to pass context without relying on globals. Pass `NULL` if not needed. |

**Without user data:**

```c
#include "ecewo.h"
#include <stdio.h>
#include "ecewo-session.h"
#include "ecewo-cors.h"
#include "db.h"

void on_shutdown(void *user_data) {
  db_cleanup();
  cors_cleanup();
  session_cleanup();
}

int main(void) {
  ecewo_app_t *app = ecewo_create();
  if (!app) {
    fprintf(stderr, "Failed to initialize server\n");
    return -1;
  }

  session_init();
  cors_init(&cors);
  db_init();

  if (ecewo_atexit(app, on_shutdown, NULL) != 0) {
    fprintf(stderr, "Failed to register shutdown callback\n");
    return -1;
  }

  ECEWO_GET(app, "/users", get_users);
  ECEWO_POST(app, "/users", create_user);

  if (ecewo_listen(app, 3000) != 0) {
    fprintf(stderr, "Failed to start server\n");
    return -1;
  }

  return 0;
}
```

**With user data:**

```c
#include "ecewo.h"
#include <stdio.h>
#include "db.h"

typedef struct {
  db_pool_t *pool;
  cache_t *cache;
} app_ctx_t;

void on_shutdown(void *user_data) {
  app_ctx_t *ctx = (app_ctx_t *)user_data;
  db_pool_destroy(ctx->pool);
  cache_destroy(ctx->cache);
  free(ctx);
}

int main(void) {
  ecewo_app_t *app = ecewo_create();
  if (!app)
    return -1;

  app_ctx_t *ctx = malloc(sizeof(app_ctx_t));
  ctx->pool  = db_pool_create("postgres://localhost/mydb", 10);
  ctx->cache = cache_create(1024);

  if (ecewo_atexit(app, on_shutdown, ctx) != 0) {
    fprintf(stderr, "Failed to register shutdown callback\n");
    return -1;
  }

  ECEWO_GET(app, "/users", get_users);
  ECEWO_POST(app, "/users", create_user);

  if (ecewo_listen(app, 3000) != 0) {
    fprintf(stderr, "Failed to start server\n");
    return -1;
  }

  return 0;
}
```

> [!WARNING]
>
> Do not place cleanup code after `ecewo_run()`. By the time `ecewo_run()` returns, the event loop has already been freed.

## Intentional Shutdown

To trigger shutdown from inside a route handler, call `ecewo_shutdown()` via `req->app`:

```c
void shutdown_handler(ecewo_request_t *req, ecewo_response_t *res) {
	ecewo_send_text(res, ECEWO_OK, "Server is shutting down");
	ecewo_shutdown(req->app);
}
```

This is safe to call from within a handler. The server finishes the current request normally, then proceeds with the shutdown sequence.

> [!NOTE]
>
> **NOTE FOR PLUGIN DEVELOPMENT:**
>
> Processes that check `ecewo_is_running()` do not need `ecewo_atexit` to receive the stop signal. `ecewo_is_running()` returns `false` as soon as shutdown begins, before any cleanup runs.
>
> Additionally, `ecewo_increment_async_work()` / `ecewo_decrement_async_work()` should be used to ensure a graceful shutdown. The async-work counter is a process-wide barrier (see [Multiple Apps](./15.multi-app.md)), pending work blocks the event loop from exiting regardless of which app initiated it.
