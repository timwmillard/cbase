# Writing FFI Bindings

This page is for people wrapping ecewo in another language (C++, Nim, Python via cffi, Lua, Zig, etc.). It documents the conventions you can rely on, the function pointer signatures you need to marshal, and the pitfalls that are specific to a C library built around a single-threaded event loop.

If you are writing application code in C, you do not need this page. Read `01.getting-started.md` instead.

## Build configuration

Build ecewo as a shared library:

```sh
cmake -B build -DECEWO_BUILD_SHARED=ON
cmake --build build
```

This produces `libecewo.so` / `libecewo.dylib` / `ecewo.dll` plus an import library on Windows. The library is built with `C_VISIBILITY_PRESET hidden`, so only symbols annotated `ECEWO_EXPORT` are exported. You can rely on this list staying stable within a major version.

Two headers are relevant:

- `include/ecewo.h`: The single public header. Everything you need to call.
- `<build-dir>/include/ecewo-export.h`: Generated, contains the `ECEWO_EXPORT` macro. Only needed if you compile C/C++ against ecewo; ctypes/cffi/dlopen-style bindings can ignore it.

ecewo also pulls in libuv and llhttp transitively. With `ECEWO_BUILD_SHARED=ON` both are built as shared libraries and will sit next to `libecewo.*`. Most bindings do not need to expose them - you should not have to load `libuv` directly unless you are integrating with the event loop.

> For a binding you ship to others, prefer the **self-contained build** (`-DECEWO_BUILD_DIST=ON`): it produces a single `libecewo` with libuv and llhttp bundled inside, so there is exactly one file to locate and load. If you also want to bind plugins, see [19.binding-plugins.md](19.binding-plugins.md).

## ABI surface

Conventions, all enforceable by a simple `nm`/`dumpbin` of the shared library:

- **Public symbols start with `ecewo_` or `ECEWO_`.** These are the only symbols you may link against. Their signatures and behavior are stable within a major version.
- **Internal symbols start with `ecewo__`** (double underscore). They are not exported from the shared library and may change or disappear in any release. Do not depend on them.
- **All public functions use the standard C calling convention** (cdecl on Windows). No fastcall / stdcall.
- **No exceptions, no longjmp, no signals raised from inside ecewo calls.** Errors are returned. See [Error reporting](#error-reporting).
- The header is wrapped in `extern "C"`, so it is C++-safe.

A short way to enumerate the API surface from a script:

```sh
nm -D --defined-only libecewo.so | awk '{print $3}' | grep '^ecewo_'
```

## Opaque handles

Every type you pass around is an opaque pointer:

| C type                       | Purpose                              |
| ---------------------------- | ------------------------------------ |
| `ecewo_app_t *`              | Application instance                 |
| `ecewo_route_t *`            | Route builder (transient)            |
| `ecewo_request_t *`          | Incoming request                     |
| `ecewo_response_t *`         | Outgoing response                    |
| `ecewo_client_t *`           | Underlying TCP client                |
| `ecewo_timer_t *`            | Timer handle                         |
| `ecewo_arena_t *`            | Arena allocator                      |
| `ecewo_takeover_config_t *`  | Connection-takeover config           |

Treat these as `void *` in your binding. Do not declare the underlying structs, do not read their fields, and do not assume any size. Every property is reached via an `ecewo_*` accessor:

```c
const char *method = ecewo_req_method(req);
const char *path = ecewo_req_path(req);
const uint8_t *body = ecewo_req_body(req);
size_t body_len = ecewo_req_body_len(req);
```

Status codes are an enum (`ecewo_status_t`) whose underlying type is the platform's `int`. You can wrap each value individually, or just pass integers - ecewo accepts any `int` for the status parameter of `ecewo_send`.

## Function pointer types

These are the signatures you must be able to produce from the host language:

```c
typedef void (*ecewo_handler_t) (ecewo_request_t *req, ecewo_response_t *res);
typedef void (*ecewo_middleware_t) (ecewo_request_t *req, ecewo_response_t *res, ecewo_next_t next);
typedef void (*ecewo_next_t) (ecewo_request_t *req, ecewo_response_t *res);

typedef void (*ecewo_timer_cb_t) (void *user_data);
typedef void (*ecewo_spawn_handler_t) (void *context);                     // work_fn
typedef void (*ecewo_spawn_done_t) (ecewo_response_t *res, void *context); // done_fn (res is NULL for background tasks)

typedef void (*ecewo_body_data_cb_t) (ecewo_request_t *req, const uint8_t *data, size_t len);
typedef void (*ecewo_body_end_cb_t) (ecewo_request_t *req, ecewo_response_t *res);
```

For connection takeover, the callbacks follow libuv signatures (`uv_alloc_cb`, `uv_read_cb`, `uv_close_cb`) and are passed in via `void *`:

```c
void ecewo_takeover_config_set_alloc_cb(ecewo_takeover_config_t*, void *alloc_cb);
void ecewo_takeover_config_set_read_cb (ecewo_takeover_config_t*, void *read_cb);
void ecewo_takeover_config_set_close_cb(ecewo_takeover_config_t*, void *close_cb);
```

There is no `user_data` parameter on `ecewo_handler_t` or `ecewo_middleware_t`. This matters.

## The closure problem

Handlers and middleware do not receive a `user_data` argument. That means you cannot directly hand ecewo a closure - every route would route through the same C function pointer and lose its identity.

There are three common workarounds. Pick whichever fits the host language:

1. **Generate a per-route C trampoline at runtime.** Use libffi (or your language's equivalent) to allocate one trampoline per route. The trampoline captures a handler ID and dispatches to a single C-side router function that looks up the host-language callable. This is the cleanest model for languages with closures (Python, Lua, Ruby).

2. **Use the request path as the key.** Inside one shared C trampoline, look up the host handler by `ecewo_req_path(req)` (or by the registered pattern). Works well if your binding stores its own route table; pay attention to dynamic segments.

3. **Stash state via the app or request, not the function.** For middleware that needs configuration, store the config on the app once at registration (`ecewo_set_app_data(app, key, cfg)`) and retrieve it inside the single trampoline (`ecewo_get_app_data(ecewo_req_app(req), key)`). Per-request state goes on the request context (`ecewo_context_set` / `ecewo_context_get`).

The `void *user_data` / `void *context` parameters on timers, spawn callbacks, and takeover callbacks **do** let you pass through a host-language object pointer. Use them.

## Threading model

ecewo runs on a single libuv event loop. Understand the threading rules before you wrap it.

- **Handlers, middleware, timer callbacks, body callbacks, spawn `done_fn`, and takeover `read_cb` / `close_cb` all run on the event-loop thread.**
- **`ecewo_spawn` `work_fn` runs on a libuv worker thread.** Do not call any other `ecewo_*` function from inside `work_fn` - produce a result and let `done_fn` send it.
- **Most `ecewo_*` functions are not thread-safe.** In particular, `ecewo_send*`, `ecewo_header_set`, `ecewo_context_*`, `ecewo_route_*`, `ecewo_use`, `ecewo_timeout`, and `ecewo_clear_timer` must be called from the loop thread.
- **Configuration setters (`ecewo_set_*`) must be called before `ecewo_listen()` / `ecewo_bind()`.**
- **`ecewo_shutdown(app)` is safe to call from inside a handler.** Cross-thread shutdown should go through `uv_async_send`; obtain the loop via `ecewo_get_loop()`.

If your host language has its own runtime (Python GIL, Node.js event loop, BEAM scheduler), the rule is: ecewo's loop must own the calls. Either run ecewo's loop on a dedicated OS thread and shuttle work back to the host runtime via your own mechanism, or block the host's main thread on `ecewo_run()` and treat ecewo as the outermost loop.

If you build long-running background work that the event loop should wait for, balance every `ecewo_increment_async_work()` with a matching `ecewo_decrement_async_work()`. The counter is process-wide and prevents `ecewo_run()` from returning while work is in flight.

## Memory and lifetimes

ecewo uses arenas. There are three lifetimes you care about:

| Arena                                | Lifetime                                  | How to get it                          |
| ------------------------------------ | ----------------------------------------- | -------------------------------------- |
| Request arena                        | Until the response is fully sent          | `ecewo_req_arena(req)` / `ecewo_res_arena(res)` |
| App arena                            | Until the app is shut down                | `ecewo_app_arena(app)`                 |
| Pool arena (scratch)                 | Caller-controlled, returned to a pool     | `ecewo_arena_borrow()` / `ecewo_arena_return()` |

Allocation helpers all take an arena:

```c
void *ecewo_alloc (ecewo_arena_t*, size_t);
void *ecewo_realloc (ecewo_arena_t*, void *oldptr, size_t oldsz, size_t newsz);
char *ecewo_strdup (ecewo_arena_t*, const char*);
void *ecewo_memdup (ecewo_arena_t*, void*, size_t);
char *ecewo_sprintf (ecewo_arena_t*, const char *fmt, ...);
```

You do not free individual allocations. The arena is released as a whole when its lifetime ends.

Rules for binding authors:

- **Pointers returned by ecewo are arena-owned.** `ecewo_param`, `ecewo_query`, `ecewo_header_get`, `ecewo_req_method`, `ecewo_req_path`, `ecewo_req_body`; all valid only until the response is sent. If you need to hand a string to the host language, copy it into a host-managed buffer before the handler returns.
- **`ecewo_send*` copies the body.** The caller's buffer does not need to outlive the call. This makes it safe to pass strings out of garbage-collected languages without pinning.
- **`ecewo_req_body` is `const uint8_t *`, not a C string.** It is not null-terminated. Always pair it with `ecewo_req_body_len`.
- **Do not call `free()` on anything ecewo returns.** Do not pass pointers from one arena into another after that arena has been released.
- **`ecewo_arena_borrow` / `ecewo_arena_return` must be balanced.** Use this for scratch work outside a request.

## Strings and encoding

- Strings passed to and returned from ecewo are byte strings. ecewo does not transcode.
- The framework assumes UTF-8 throughout. Header lookup is case-insensitive (names are lowercased at parse time, but you can pass any case to `ecewo_header_get`).
- Request body bytes are raw (`uint8_t*`), with no encoding assumption.
- `ecewo_set_listen_address` requires a numeric IPv4 or IPv6 string; hostnames are not resolved.

## Error reporting

ecewo uses two conventions:

- Functions that can fail return `int`: `0` for success, `-1` (or another non-zero value) for failure. Examples: `ecewo_bind`, `ecewo_listen`, `ecewo_atexit`, `ecewo_use`, `ecewo_spawn`, `ecewo_timeout_request`, `ecewo_connection_takeover`.
- Functions that return a handle return `NULL` on failure. Examples: `ecewo_create`, `ecewo_route_new`, `ecewo_timeout`, `ecewo_interval`, `ecewo_takeover_config_new`.

Functions that produce a response (`ecewo_send*`, `ecewo_redirect`, `ecewo_header_set`) return `void` and either succeed or log an error. Specifically, `ecewo_header_set` silently no-ops on reserved framing headers (`Content-Length`, `Transfer-Encoding`, `Connection`, `Host`, `Date`) - your binding should either filter those out before calling or document the behavior.

ecewo does not call `exit()` or `abort()` from inside public APIs in v4. Earlier versions did; if you are reading older binding code that handles SIGABRT, it is no longer needed.

## What to wrap and what to skip

**Wrap:**

- The full `ecewo_*` function surface listed in `ecewo.h`.
- The `ecewo_status_t` enum values you intend to expose. Or just expose them as integer constants.
- The `ecewo_method_t` enum, needed for `ecewo_route_new`.

**Use the function form, not the macro:**

The `ECEWO_GET` / `ECEWO_POST` / ... macros are C macros and not callable from FFI. Use the route builder instead - it is designed for this case:

```c
ecewo_route_t *r = ecewo_route_new(app, ECEWO_METHOD_GET, "/users/:id");
ecewo_route_middleware(r, auth_mw);  // optional, called 0+ times
ecewo_route_handler(r, get_user);    // finalizes the route
```

See `02.defining-routes.md` for the full builder reference.

**Skip from FFI:**

- The `ECEWO_DA_*` / `ECEWO_SB_*` dynamic-array and string-builder macros. They are C-only ergonomic helpers and have no value across an FFI boundary - your host language has its own containers.
- The `ECEWO__*` internal macros (e.g. `ECEWO__DA_INIT_CAP`, `ECEWO__CAST_PTR`).
- Type forwards like `ecewo_arena_s`, `ecewo_request_s` - they are intentionally incomplete.

## Lifecycle

The standard call sequence:

```c
ecewo_app_t *app = ecewo_create();             // allocate app
// (optional) ecewo_set_*(app, ...) for configuration
// (optional) ecewo_use(app, path, mw)
// register routes via ecewo_route_new/middleware/handler
ecewo_atexit(app, on_shutdown, user_data);     // optional
ecewo_listen(app, 3000);                       // blocks until shutdown
// (control returns here after ecewo_shutdown(app) or a signal)
```

If you need to register additional libuv handles or start timers before the loop runs, split `ecewo_listen` into:

```c
ecewo_bind(app, 3000);   // bind but do not run
// ... add timers, handles
ecewo_run();             // enter the loop
```

`ecewo_run` is a process-wide singleton; calling it from multiple threads is not supported.

`ecewo_atexit` callbacks run during `ecewo_shutdown`, before the app is unregistered. This is where bindings should release per-app host state (close native handles, decrement reference counts on stored callables, etc.). The `user_data` you registered is passed through unchanged.

## libuv integration

`ecewo_get_loop()` returns the underlying `uv_loop_t *` (typed as `void *` to avoid forcing `uv.h` on every binding). With it you can:

- Wake the loop from another thread via `uv_async_t`.
- Attach additional file watchers, signal handlers, or sockets.
- Drive integrations that already speak libuv.

Using the loop directly puts the burden of thread safety and lifetime correctness on you. If you only need to offload blocking work, prefer `ecewo_spawn`.

For raw socket protocols (WebSocket, MQTT), the supported path is `ecewo_connection_takeover`: send your upgrade response, hand ecewo a config struct with libuv callbacks, and ecewo stops reading from the socket while keeping its bookkeeping intact. Pair every takeover with `ecewo_takeover_close_socket` on close.

## Versioning and SOVERSION

The shared library is built with `SOVERSION` equal to the major version (currently `4`). Linking against `libecewo.so.4` guarantees a stable ABI for the lifetime of the v4.x line: no removed or renamed exports, no signature changes, no semantic regressions on documented behavior. New exports may be added.

Bumping the major version (v4 → v5) is the only point at which we change the ABI. Bindings should pin the SOVERSION they were compiled against.

## Minimal binding sketch

A barely-fleshed-out C++ sketch, showing the trampoline pattern for closures. Adapt to whatever language you are targeting.

```cpp
#include "ecewo.h"
#include <functional>
#include <unordered_map>
#include <string>

using HostHandler = std::function<void(ecewo_request_t*, ecewo_response_t*)>;

// One table keyed by route pattern; populated when the host registers a route.
static std::unordered_map<std::string, HostHandler> g_handlers;

// Single C entry point that every ecewo handler routes through.
extern "C" void cpp_trampoline(ecewo_request_t* req, ecewo_response_t* res) {
  // In a real binding you'd key by the registered pattern, not the live path.
  // For simplicity, store the pattern on the request via ecewo_context_set at registration.
  auto* key = static_cast<const char*>(ecewo_context_get(req, "__route_key"));
  auto it = g_handlers.find(key ? key : "");
  if (it != g_handlers.end()) it->second(req, res);
  else ecewo_send_text(res, ECEWO_NOT_FOUND, "no handler");
}

void host_register(ecewo_app_t* app, ecewo_method_t m, const std::string& path, HostHandler h) {
  g_handlers[path] = std::move(h);
  ecewo_route_t* r = ecewo_route_new(app, m, path.c_str());
  ecewo_route_handler(r, &cpp_trampoline);
}
```

For closure-capturing languages without C++-style static globals (Python, Lua), allocate per-route trampolines with libffi and pass each one as the handler. That gives every route its own unique C function pointer, so the trampoline can carry a route ID directly without a path lookup.
