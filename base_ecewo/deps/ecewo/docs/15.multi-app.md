# Multiple Apps in One Process

ecewo supports running multiple `ecewo_app_t` instances in the same process. They share a single event loop; the same model Express.js uses, where you can `express()` more than once and `app.listen()` each on a different port.

## Overview

Every call to `ecewo_create()` returns an independent app with its own routes, middleware, configuration, listening socket, and `ecewo_atexit()` callback. The first call to `ecewo_create()` lazily initializes the process-level runtime: the event loop, signal handlers, and the global async-work counter. Every subsequent app attaches to that same runtime.

Common reasons to use more than one app:

- Public traffic and an admin/health port in the same binary.
- Different middleware stacks bound to different ports (e.g. CORS-enabled API on 8080, internal-only RPC on 9000).
- Test harnesses that boot two apps to verify isolation.

## Example

```c
#include "ecewo.h"

static void public_root(ecewo_request_t *req, ecewo_response_t *res) {
  (void)req;
  ecewo_send_text(res, 200, "public");
}

static void admin_health(ecewo_request_t *req, ecewo_response_t *res) {
  (void)req;
  ecewo_send_text(res, 200, "ok");
}

int main(void) {
  ecewo_app_t *pub = ecewo_create();
  ecewo_app_t *admin = ecewo_create();
  if (!pub || !admin)
    return 1;

  ECEWO_GET(pub, "/", public_root);
  ECEWO_GET(admin, "/health", admin_health);

  if (ecewo_bind(pub, 8080) != 0) return 1;
  if (ecewo_bind(admin, 9000) != 0) return 1;

  // Runs the shared loop. Blocks until every app has been shut down
  // (or SIGINT/SIGTERM is received).
  ecewo_run();
  return 0;
}
```

## Lifecycle

```
ecewo_create()  ── first call ──> runtime lazy-init (loop, signals, atexit)
                └── subsequent ──> attach app to existing runtime

ecewo_bind(app, port)             attach a TCP listener to the shared loop

ecewo_run()                       block on the shared loop
                                  returns when every app is shut down

ecewo_shutdown(app)               close that app's listener, drain its
                                  connections, fire its atexit callback,
                                  unregister it from the runtime

SIGINT / SIGTERM                  shut down every registered app
```

`ecewo_run()` takes no arguments, it runs the process-level loop, not a per-app one. At least one app must have been bound (via `ecewo_bind()` or `ecewo_listen()`) before calling it; otherwise the loop has nothing to do and the call returns with an error logged. Recursive calls (e.g. from a handler) are no-ops.

## What is shared, what is per-app

| Resource                          | Scope         |
|-----------------------------------|---------------|
| Event loop (`ecewo_get_loop()`)   | Process       |
| Signal handlers (SIGINT/SIGTERM)  | Process       |
| Async-work counter (`ecewo_increment_async_work` / `decrement`) | Process |
| Timers (`ecewo_timeout`/`ecewo_interval`) | Process |
| Background work (`ecewo_spawn`)   | Process       |
| TCP listener (`ecewo_bind`)       | Per-app       |
| Active connections                | Per-app       |
| Routes / middleware               | Per-app       |
| App configuration (`ecewo_set_*`) | Per-app       |
| `ecewo_atexit` callback           | Per-app       |
| Shutdown timeout                  | Per-app       |

Plugin-author APIs that take no `ecewo_app_t *` (`ecewo_get_loop`, `ecewo_increment_async_work`, `ecewo_decrement_async_work`) refer to the shared runtime by design - the loop is a process singleton, and an async-work barrier that doesn't span apps would be incorrect.

## Caveats

- **One port per app.** If you need multiple ports for the same handler set, create multiple apps that share handler functions.
- **`ecewo_run()` is single-threaded.** Like libuv itself, the loop must be driven by exactly one thread.
- **Pointers are dangling after `ecewo_run()` returns.** Once shutdown completes, every app's internal state has been freed and the `ecewo_app_t` struct itself has been freed. Don't keep app pointers after `ecewo_run()` returns.
- **Shutdown order is not guaranteed.** A `SIGINT` triggers shutdown of every registered app concurrently; their `ecewo_atexit` callbacks may run in any order.
- **Worker mode (`ECEWO_WORKER=1`).** Signal handlers are not installed in worker mode regardless of how many apps you create - the supervising process is expected to deliver shutdown signals.
