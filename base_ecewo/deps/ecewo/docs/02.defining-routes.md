# Defining Routes

We can create our routes with these macros:

- `ECEWO_GET()` for GET routes,
- `ECEWO_POST()` for POST routes,
- `ECEWO_PUT()` for PUT routes,
- `ECEWO_DELETE()` for DELETE routes,
- `ECEWO_PATCH()` for PATCH routes,
- `ECEWO_OPTIONS()` for OPTIONS routes,
- `ECEWO_QUERY()` for QUERY routes.

All route macros take `app` as their first parameter, followed by the path and handler(s).

They must be defined after creating the app in the entry point, which is the `int main(void)` function in the `main.c` file.

```c
#include "ecewo.h"
#include "our_handlers.h"
#include <stdio.h>

int main(void) {
  ecewo_app_t *app = ecewo_create();
  if (!app) {
    fprintf(stderr, "Failed to initialize server\n");
    return 1;
  }

  // Define the routes
  ECEWO_DELETE(app, "/delete", sample_delete_handler);
  ECEWO_PUT(app, "/update", sample_update_handler);
  ECEWO_PATCH(app, "/edit", sample_edit_handler);
  ECEWO_POST(app, "/create", sample_create_handler);
  ECEWO_GET(app, "/", sample_get_handler);

  if (ecewo_listen(app, 3000) != 0) {
    fprintf(stderr, "Failed to start server\n");
    return 1;
  }

  return 0;
}
```

> [!TIP]
>
> We also can define all of our routes in one function and call it once after creating the app in main function:

```c
#include "ecewo.h"
#include "our_handlers.h"
#include <stdio.h>

void register_all_routes(ecewo_app_t *app) {
  // Define the routes
  ECEWO_DELETE(app, "/delete", sample_delete_handler);
  ECEWO_PUT(app, "/update", sample_update_handler);
  ECEWO_PATCH(app, "/edit", sample_edit_handler);
  ECEWO_POST(app, "/create", sample_create_handler);
  ECEWO_GET(app, "/", sample_get_handler);
}

int main(void) {
  ecewo_app_t *app = ecewo_create();
  if (!app) {
    fprintf(stderr, "Failed to initialize server\n");
    return 1;
  }

  // Register all the routes
  register_all_routes(app);

  if (ecewo_listen(app, 3000) != 0) {
    fprintf(stderr, "Failed to start server\n");
    return 1;
  }

  return 0;
}
```

## Dynamic Parameters

Use `:param` syntax for dynamic segments:

```c
ECEWO_GET(app, "/users/:id", user_get_handler);
ECEWO_POST(app, "/users/:id", user_post_handler);
```

## Wildcard Routes

```c
#include "ecewo.h"
#include <stdio.h>

void catch_all(ecewo_request_t *req, ecewo_response_t *res) {
  ecewo_send_text(res, 404, "Not Found");
}

int main(void) {
  ecewo_app_t *app = ecewo_create();
  if (!app) {
    fprintf(stderr, "Failed to initialize server\n");
    return 1;
  }

  // Matches any route not already defined
  ECEWO_GET(app, "*", catch_all);

  if (ecewo_listen(app, 3000) != 0) {
    fprintf(stderr, "Failed to start server\n");
    return 1;
  }

  return 0;
}
```

## Builder Pattern

The `ECEWO_*` macros are variadic and resolve the function list at compile time, so they cannot be used when the handler or middleware array is built dynamically, and variadic C macros cannot be called from foreign language bindings at all. For those cases (registering routes from data at runtime, writing plugins, or calling from an FFI binding) ecewo exposes the underlying builder API:

- `ecewo_route_new(app, method, path)`: start a new route. Returns an opaque `ecewo_route_t *`.
- `ecewo_route_middleware(route, fn)`: attach a middleware to the chain. Call once per middleware, in order.
- `ecewo_route_handler(route, fn)`: finalize the route with its handler and register it with the router.

The builder is arena-allocated, so there is nothing to free. Once `ecewo_route_handler` is called the builder must not be reused.

The `method` argument uses the `ecewo_method_t` enum:

```c
ECEWO_METHOD_GET
ECEWO_METHOD_POST
ECEWO_METHOD_PUT
ECEWO_METHOD_PATCH
ECEWO_METHOD_DELETE
ECEWO_METHOD_HEAD
ECEWO_METHOD_OPTIONS
```

### Minimal example

```c
ecewo_route_t *r = ecewo_route_new(app, ECEWO_METHOD_GET, "/users/:id");
ecewo_route_handler(r, get_user_handler);
```

This is equivalent to:

```c
ECEWO_GET(app, "/users/:id", get_user_handler);
```

### With middleware

Add middleware before the handler. The chain runs in registration order, then the final handler:

```c
ecewo_route_t *r = ecewo_route_new(app, ECEWO_METHOD_POST, "/admin/settings");
ecewo_route_middleware(r, auth_middleware);
ecewo_route_middleware(r, validate_settings);
ecewo_route_handler(r, settings_handler);
```

This is equivalent to:

```c
ECEWO_POST(app, "/admin/settings", auth_middleware, validate_middleware, settings_handler);
```

### Registering routes from data

Because the builder takes plain function pointers and the path as a `const char *`, you can drive registration from a table:

```c
typedef struct {
  ecewo_method_t method;
  const char *path;
  ecewo_handler_t handler;
} route_def_t;

static const route_def_t routes[] = {
  { ECEWO_METHOD_GET, "/", home_handler },
  { ECEWO_METHOD_GET, "/users/:id", get_user },
  { ECEWO_METHOD_POST, "/users", create_user },
};

for (size_t i = 0; i < sizeof(routes) / sizeof(routes[0]); i++) {
  ecewo_route_t *r = ecewo_route_new(app, routes[i].method, routes[i].path);
  ecewo_route_handler(r, routes[i].handler);
}
```

> [!NOTE]
>
> There is also a lower-level function that the `ECEWO_*` macros call directly:
>
> ```c
> void ecewo_route_register(ecewo_app_t *app, ecewo_method_t method,
>                           const char *path, void **fns, int count);
> ```
>
> `fns` is a flat array of `[middleware0, ..., middlewareN, handler]` and `count` is the total number of elements. The builder API wraps this function and is preferred in new code. Reach for `ecewo_route_register` directly only when you already have a pre-built function-pointer array:
>
> ```c
> void *fns[] = { auth_middleware, get_user_handler };
> ecewo_route_register(app, ECEWO_METHOD_GET, "/users/:id", fns, 2);
> ```
