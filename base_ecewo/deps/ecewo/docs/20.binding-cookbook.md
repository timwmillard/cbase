# Worked Example: Building and Publishing an ecewo Binding

This page is a tutorial. It walks through every step of taking ecewo (plus a
plugin) and turning it into an installable package in another language —
using Python as the running example, but with notes throughout for how to
adapt to Ruby, Lua, Node.js / N-API, Zig, Nim, or anything else with a C FFI.

It is the practical companion to two reference documents:

- [`17.ffi-bindings.md`](17.ffi-bindings.md) — the ABI contract, function-
  pointer signatures, threading model, memory rules, error conventions.
- [`19.binding-plugins.md`](19.binding-plugins.md) — how plugins fit on top
  of a core binding.

Those two are the spec. **This page is the cookbook.** Read 16 and 19 first
if you haven't.

The reference implementation is [`ecewopy`](https://github.com/savashn/ecewopy)
— a Python binding for ecewo + the `ecewo-fs` plugin, packaged for PyPI.
Code linked below points there.

## What we're building

The end state: a user runs

```sh
pip install ecewopy
```

…and gets a Python package that wraps `libecewo.so` and `libecewo-fs.so`,
with this API:

```python
import ecewopy
import ecewopy.fs as fs

fs.init()
app = ecewopy.App()
app.get("/", lambda req, res: res.send_text(ecewopy.OK, "hello\n"))
app.listen(3000)
```

The user does not install CMake. They do not clone the ecewo repo. They do
not have a C compiler. They get a prebuilt wheel that contains the
shared libraries, drops them inside their Python environment, and never
touches anything system-wide.

Everything below is how to get from "ecewo is a C library on GitHub" to
"users `pip install` it and it just works".

## Choosing a distribution strategy

**Your binding builds the C library itself, in CI.** Your build system
clones ecewo (and each plugin) at a pinned git ref, compiles them fresh for
each platform you support, and bundles the resulting `.so` files inside
your platform-specific wheel. The user downloads a single artifact; nothing
is compiled on their machine.

This is the model `ecewopy` uses, and what the rest of this document
assumes. ecewo does not publish prebuilt binaries on GitHub Releases — the
source on git is the only distribution channel. There is nothing for your
binding to download at build time; it clones, compiles, packages.

The tradeoff is slightly longer CI builds (you compile ecewo from source
each time you build a wheel), which doesn't matter for end users — wheel
builds happen in CI on tag pushes, not on `pip install`.

## Package layout

```
ecewopy/
├── pyproject.toml              # build backend (scikit-build-core), project metadata
├── CMakeLists.txt              # FetchContent ecewo + ecewo-fs; build shared; install
├── README.md
├── LICENSE
├── .gitignore
├── src/
│   └── ecewopy/
│       ├── __init__.py         # public API: App, Request, Response, status codes
│       ├── _loader.py          # finds and dlopens the bundled .so files
│       ├── _core.py            # ctypes binding for ecewo
│       ├── fs.py               # ctypes binding for ecewo-fs
│       └── _libs/              # populated at build time; gitignored
│           ├── libecewo.so*    # bundled into the wheel
│           └── libecewo-fs.so*
├── examples/server.py          # runnable demo
├── tests/
│   ├── conftest.py             # daemon-thread server fixture
│   ├── test_smoke.py
│   └── test_fs.py
└── .github/workflows/python-wheels.yml   # cibuildwheel + PyPI publishing
```

The split keeps source-of-truth in `src/ecewopy/`. `_libs/` is empty in the
repo; CI populates it. The wheel that ships to users contains everything
except `_libs/.gitkeep`.

## The build system: pyproject.toml + CMakeLists.txt

Modern Python C-extension packaging uses two coordinated files:

`pyproject.toml` declares **what** to build — package name, version, Python
versions, dependencies, and the build backend. We use
[`scikit-build-core`](https://scikit-build-core.readthedocs.io/), which is
the standard backend for CMake-driven Python packages.

```toml
[build-system]
requires = ["scikit-build-core>=0.9"]
build-backend = "scikit_build_core.build"

[project]
name = "ecewopy"
version = "0.1.0"
requires-python = ">=3.11"

[tool.scikit-build]
wheel.packages = ["src/ecewopy"]
# Files our CMake `install()` creates under CMAKE_INSTALL_PREFIX end up here
# inside the wheel, next to _loader.py:
wheel.install-dir = "ecewopy/_libs"
# Filter out cruft that subprojects (libuv, llhttp) install: their headers,
# .a archives, pkg-config files, CMake configs.
wheel.exclude = [
  "include/**", "share/**",
  "lib/**/*.a", "lib/**/cmake/**", "lib/**/pkgconfig/**",
  "lib64/**/*.a", "lib64/**/cmake/**", "lib64/**/pkgconfig/**",
]
```

`CMakeLists.txt` declares **how** to build — fetch ecewo, build it in DIST
mode (libuv and llhttp bundled inside libecewo so we ship one self-contained
`.so`), pull in plugins, and install only the runtime files:

```cmake
cmake_minimum_required(VERSION 3.15)
project(ecewopy LANGUAGES C)

# Lets `option(...)` calls in fetched subprojects respect normal variables
# we set in this scope.
cmake_policy(SET CMP0077 NEW)

include(FetchContent)

# Forwarded into ecewo's CMakeLists. Normal variables, not CACHE: CACHE
# alone gets overridden by `option(... OFF)` in the fetched subproject.
set(ECEWO_BUILD_DIST       ON)
set(ECEWO_BUILD_TESTS      OFF)
set(ECEWO_FS_BUILD_SHARED  ON)

FetchContent_Declare(
  ecewo
  GIT_REPOSITORY https://github.com/ecewo/ecewo.git
  GIT_TAG        v4
  GIT_SHALLOW    TRUE
)
FetchContent_MakeAvailable(ecewo)
ecewo_add(fs@dev)

# Rpath so libecewo-fs.so finds its libecewo.so neighbour in the wheel.
foreach(_t ecewo ecewo-fs)
  set_target_properties(${_t} PROPERTIES
    INSTALL_RPATH "$<IF:$<BOOL:${APPLE}>,@loader_path,$ORIGIN>"
    BUILD_WITH_INSTALL_RPATH ON
  )
endforeach()

install(TARGETS ecewo ecewo-fs
  LIBRARY DESTINATION .
  RUNTIME DESTINATION .)
```

Three gotchas this captures, learned from actually doing it:

- **CMP0077 NEW is required**, otherwise ecewo's `option(ECEWO_BUILD_DIST ... OFF)`
  silently resets the value we set in the parent project and you end up
  with a static archive that won't fit a shared library.
- **Use normal variables (no `CACHE`)** in the parent for the same reason —
  `set(... CACHE BOOL "" FORCE)` propagates to the cache but `option()` can
  still clobber the normal-variable view. Setting both works; setting only
  CACHE doesn't.
- **DIST mode bundles libuv + llhttp into libecewo** at link time, so the
  wheel ships one `.so` for the core, not three. Without it (using
  `ECEWO_BUILD_SHARED` instead), libecewo would `NEEDED libuv.so.1` and
  you'd have to also ship libuv and libllhttp with matching rpaths.

## Loading the shared libraries

The plugin's `.so` has undefined `ecewo_*` symbols that must resolve to the
**same** libecewo as the rest of the binding — see
[`19.binding-plugins.md`](19.binding-plugins.md#loading-plugins-from-your-binding).

In Python the recipe is two lines:

```python
import ctypes, os, sys

_LIB_DIR = os.path.join(os.path.dirname(__file__), "_libs")

def _name(stem):
    if sys.platform == "darwin": return f"lib{stem}.dylib"
    if sys.platform == "win32":  return f"{stem}.dll"
    return f"lib{stem}.so"

def load_core():
    return ctypes.CDLL(os.path.join(_LIB_DIR, _name("ecewo")),
                       mode=ctypes.RTLD_GLOBAL)   # <-- this part is important

def load_plugin(stem):
    return ctypes.CDLL(os.path.join(_LIB_DIR, _name(stem)))
```

`RTLD_GLOBAL` on the core load is what makes the plugin loadable later. In
Ruby use `Fiddle::RTLD_GLOBAL`. In Lua use the loader's equivalent flag. In
Node/N-API the plugin is itself a native addon and gets resolved through
node's loader. In Zig/Nim the linker baking the binding produces a single
shared object — the issue doesn't come up at runtime, only at build time.

## Marshalling the C ABI

Every ecewo type is opaque (see [`17.ffi-bindings.md`](17.ffi-bindings.md#opaque-handles)).
The binding treats them all as `void *`:

```python
ecewo_app_p      = c_void_p
ecewo_request_p  = c_void_p
ecewo_response_p = c_void_p
ecewo_arena_p    = c_void_p
```

Function signatures get declared once and reused:

```python
def _bind(fn, restype, *argtypes):
    fn.restype, fn.argtypes = restype, list(argtypes)

_bind(_lib.ecewo_create,        ecewo_app_p)
_bind(_lib.ecewo_listen,        c_int, ecewo_app_p, c_uint16)
_bind(_lib.ecewo_route_new,     c_void_p, ecewo_app_p, c_int, c_char_p)
_bind(_lib.ecewo_route_handler, None, c_void_p, HANDLER_FN)
_bind(_lib.ecewo_send_text,     None, ecewo_response_p, c_int, c_char_p)
# ...etc, one line per export you wrap
```

Status codes (`ECEWO_OK = 200`, etc.) and method enums (`ECEWO_METHOD_GET = 1`,
etc.) are plain ints — no need to wrap them as C enums. The
[full table is in the API reference](16.api-reference.md).

In Ruby this same pattern uses `Fiddle::Function.new(symbol, [arg_types], ret_type)`.
In Lua via `ffi.cdef "..."`. In Node via `napi_create_function`. The
mechanics differ; the contract is identical.

**Wrap the function-style API, not the macros.** The `ECEWO_GET` / `ECEWO_POST`
macros in `ecewo.h` are C preprocessor sugar and not callable from FFI.
Use `ecewo_route_new` / `ecewo_route_middleware` / `ecewo_route_handler`.

## The closure problem in practice

This is the design problem in every binding. ecewo's handler signature is

```c
typedef void (*ecewo_handler_t)(ecewo_request_t *req, ecewo_response_t *res);
```

— no `user_data`, no environment pointer. So you can't just register a
single C function and pass it the host-language callable. Every route needs
its own unique C function pointer.

[`17.ffi-bindings.md`](17.ffi-bindings.md#the-closure-problem) describes
three options. In a language with first-class function-pointer trampolines
(Python via ctypes, Lua via ffi.cast, Ruby via Fiddle::Closure), option 1
is by far the cleanest:

```python
HANDLER_FN = CFUNCTYPE(None, ecewo_request_p, ecewo_response_p)

def _make_handler_trampoline(py_handler):
    @HANDLER_FN
    def trampoline(req_p, res_p):
        py_handler(Request(req_p), Response(res_p))
    _keepalive.append(trampoline)
    return trampoline
```

Each `@HANDLER_FN`-decorated function produces a CFUNCTYPE instance with a
**unique C function pointer address**, captured around the Python callable
we want to dispatch to. Calling this from `ecewo_route_handler` registers
exactly that route → that handler.

The pattern generalizes: middleware uses `MIDDLEWARE_FN`, body callbacks
use their own `CFUNCTYPE`, timer callbacks use `TIMER_FN`, etc. One
trampoline per registration, all going through one tiny dispatcher.

## Two lifetime pitfalls

Both of these were caught only by actually running the binding. They are
the kind of bug that doesn't show up in single-route hello-world tests.

**1. CFUNCTYPE objects must outlive the C side's reference to them.** If
Python garbage-collects a trampoline while ecewo still holds its function
pointer, the next request segfaults. Pin every trampoline you create in a
module-level list:

```python
_keepalive = []  # holds every CFUNCTYPE we hand to ecewo

def _make_handler_trampoline(py_handler):
    @HANDLER_FN
    def trampoline(req_p, res_p): ...
    _keepalive.append(trampoline)   # <-- without this: segfault on second request
    return trampoline
```

For one-shot callbacks (a timer fires once, a `fs.read_file` completes
once), `_keepalive.remove(cb)` after the callback fires to release the
pin.

**2. `ecewo_route_new` stores the path pointer; the strdup happens later.**
This one is in the C code: `ecewo_route_new` stashes the `const char *path`
argument as-is, and only copies it inside `ecewo_route_handler` via
`route_table_add`. If the host language's bytes object that held the path
gets freed between those two calls, the router copies garbage.

In Python the naïve binding does this:

```python
def route(self, method, path, handler):
    r = _lib.ecewo_route_new(self._app, method, path.encode())  # <-- temp bytes
    _lib.ecewo_route_handler(r, _make_handler_trampoline(handler))
```

`path.encode()` produces a temporary `bytes` object that lives only for
the duration of the `ecewo_route_new` call. After the call returns, Python
can free it. The second route you register sees its path string corrupted.
Only the first route survives.

The fix is to pin path bytes for the app's lifetime:

```python
class App:
    def __init__(self):
        self._app = _lib.ecewo_create()
        self._pinned = []   # <-- pin every byte string handed to ecewo

    def route(self, method, path, handler, middleware=()):
        path_bytes = path.encode()
        self._pinned.append(path_bytes)   # <-- prevents GC
        r = _lib.ecewo_route_new(self._app, method, path_bytes)
        # ...
```

The same applies to anything else ecewo stores by pointer: prefixed paths
on `ecewo_use`, addresses on `ecewo_set_listen_address`, context keys on
`ecewo_context_set`. If a function takes `const char *` and you can't
verify it strdups immediately, pin the bytes.

This pattern is language-agnostic. Ruby has the same GC; Lua does too; Node
has it via V8. Any host language that GCs strings between calls hits this.

## Wrapping requests, responses, middleware

These wrappers are thin: they translate Python `str` ↔ C `bytes`, and copy
out any pointer ecewo returns (since arena-owned pointers are valid only
inside the handler — see [`17.ffi-bindings.md`](17.ffi-bindings.md#memory-and-lifetimes)).

```python
class Request:
    __slots__ = ("_p",)
    def __init__(self, p): self._p = p

    @property
    def method(self):
        v = _lib.ecewo_req_method(self._p)
        return v.decode() if v else ""

    @property
    def body(self):
        n = _lib.ecewo_req_body_len(self._p)
        if n == 0: return b""
        ptr = _lib.ecewo_req_body(self._p)
        return ctypes.string_at(ptr, n)   # copy out

    def param(self, key):
        v = _lib.ecewo_param(self._p, key.encode())
        return v.decode() if v else None
```

The `ctypes.string_at(ptr, n)` is the crucial step: it makes a Python
`bytes` copy before the per-request arena is freed. If you returned a
`c_char_p` directly, the user could hold it past the response and segfault.

Middleware is the same trampoline pattern as handlers, but the C
signature also takes a `next` pointer:

```python
MIDDLEWARE_FN = CFUNCTYPE(None, ecewo_request_p, ecewo_response_p, NEXT_FN)

def _make_middleware_trampoline(py_mw):
    @MIDDLEWARE_FN
    def trampoline(req_p, res_p, next_fn):
        def call_next(): next_fn(req_p, res_p)
        py_mw(Request(req_p), Response(res_p), call_next)
    _keepalive.append(trampoline)
    return trampoline
```

## Wrapping a plugin (ecewo-fs)

Per [`19.binding-plugins.md`](19.binding-plugins.md), a plugin is just
another shared library you load **after** core, then bind each exported
function the same way.

```python
# fs.py
from . import _core  # ensure libecewo is loaded RTLD_GLOBAL FIRST
from ._loader import load_plugin

_libfs = load_plugin("ecewo-fs")

FS_READ_CB = CFUNCTYPE(None, c_char_p, c_void_p, c_size_t, c_void_p)

_bind(_libfs.fs_init,      c_int)
_bind(_libfs.fs_read_file, c_int, c_char_p, c_void_p, FS_READ_CB, c_void_p)

def read_file(path, on_done, *, arena=None):
    @FS_READ_CB
    def cb(error, data, size, _user):
        try:
            if error:        on_done(error.decode(), None)
            elif size == 0:  on_done(None, b"")
            else:            on_done(None, ctypes.string_at(data, size))
        finally:
            _keepalive.remove(cb)   # one-shot

    _keepalive.append(cb)
    _libfs.fs_read_file(path.encode(), arena, cb, None)
```

Three things to notice:

- `from . import _core` is not a no-op import — it's the mechanism that
  forces libecewo to load RTLD_GLOBAL before we touch libecewo-fs.
- The arena parameter is `c_void_p`. Inside a handler, callers pass
  `req.arena_ptr` so the buffer is freed with the response. Outside a
  handler, pass `None` and ecewo-fs falls back to `malloc`. **The binding
  copies the bytes into a Python `bytes` inside the callback, so the
  caller never deals with raw pointers.**
- The callback is a one-shot; we drop it from `_keepalive` once it fires.

Adding more plugins is mechanical: copy this file, change the `_libfs`
target, bind the new symbols. No new core work required.

## Testing the binding

ecewo's `listen()` blocks on the event loop. Cross-thread `shutdown()` is
not safe (see [`17.ffi-bindings.md`](17.ffi-bindings.md#threading-model)).
The trick for tests:

1. Run `app.listen()` in a daemon thread.
2. Register a `/_/shutdown` route that calls `app.shutdown()` from inside
   the handler — that's the supported path because the handler runs on the
   loop thread.
3. The test main thread issues `urllib.urlopen("/_/shutdown")` during
   teardown.

```python
# tests/conftest.py
@pytest.fixture
def server():
    started = []
    def _start(configure):
        port = _free_port()
        app = ecewopy.App()
        configure(app)
        app.get("/_/shutdown", lambda req, res: (app.shutdown(),
                                                 res.send_text(200, "bye")))
        t = threading.Thread(target=app.listen, args=(port,), daemon=True)
        t.start()
        _wait_listening(port)
        s = SimpleNamespace(port=port, url=f"http://127.0.0.1:{port}")
        started.append((s, app, t))
        return s
    yield _start
    for s, app, t in started:
        urllib.request.urlopen(f"{s.url}/_/shutdown", timeout=2).read()
        t.join(timeout=5)
```

Each test gets a fresh `App` on a random ephemeral port, makes a few HTTP
requests against it, and shuts it down cleanly. The same approach works
in Ruby (`Thread.new { app.listen ... }`) and Lua (`coroutine.wrap` or
`copas.addthread`).

The 11-test `ecewopy` test suite uses this pattern to cover routes,
params, query strings, body echo, middleware chain order, JSON, 404, the
path-bytes lifetime regression, and the two `fs` operations. It runs in
30ms.

## Publishing

For Python the canonical channel is PyPI. The CI pattern is
[cibuildwheel](https://cibuildwheel.pypa.io/), which builds a wheel for
each `(OS, arch, Python version)` combination in a manylinux/macOS/Windows
matrix.

```yaml
# .github/workflows/python-wheels.yml
on:
  push:
    tags: ["v*"]

jobs:
  build_wheels:
    strategy:
      matrix:
        include:
          - { os: ubuntu-latest, arch: x86_64 }
          - { os: ubuntu-latest, arch: aarch64 }    # via QEMU
          - { os: macos-14,      arch: arm64 }
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-qemu-action@v3
        if: matrix.arch == 'aarch64'
      - uses: pypa/cibuildwheel@v2.21
        env:
          CIBW_BUILD: "cp311-* cp312-* cp313-*"
          CIBW_ARCHS_LINUX: ${{ matrix.arch }}
          CIBW_ARCHS_MACOS: ${{ matrix.arch }}
          CIBW_BEFORE_ALL_LINUX: yum install -y cmake3 ninja-build && ln -sf /usr/bin/cmake3 /usr/local/bin/cmake

  publish:
    needs: build_wheels
    permissions: { id-token: write }   # OIDC for PyPI Trusted Publishing
    steps:
      - uses: actions/download-artifact@v4
      - uses: pypa/gh-action-pypi-publish@release/v1
```

The user installs with `pip install ecewopy`. pip looks at pypi.org, sees
nine wheels (3 platforms × 3 Python versions) plus an sdist, picks the
matching wheel, downloads it (~500 KB), and unzips it into the user's
Python `site-packages/`. The `.so` files land at
`site-packages/ecewopy/_libs/libecewo.so` and friends. **Nothing is
installed system-wide**; no `sudo`, no `ldconfig`, no compiler.

For first-time setup:

1. Register the package name on pypi.org.
2. Configure PyPI [Trusted Publishing](https://docs.pypi.org/trusted-publishers/)
   pointing at your GitHub repo + workflow filename. No API tokens to manage.
3. Push a `v0.1.0` tag. The workflow runs, builds, publishes.

The equivalent for other languages:

| Language | Registry | CI pattern |
|---|---|---|
| Python | PyPI | cibuildwheel matrix |
| Ruby | RubyGems | rake-compiler-dock per platform |
| Lua | LuaRocks | rockspec with external dependencies |
| Node.js | npm | prebuildify / node-pre-gyp |
| Rust | crates.io | source-only by convention, users compile |
| Zig | source distribution | users `zig build` from source |
| Nim | nimble | source-only or prebuilt binaries |

## Adapting to other host languages

The four hard parts of binding ecewo are language-agnostic:

1. **Loading the core RTLD_GLOBAL, then plugins.** Every language with a
   shared-library loader exposes this. The constants have different names
   but the semantics are identical.

2. **Per-callable C trampolines for the closure problem.** Python's
   `ctypes.CFUNCTYPE`, Ruby's `Fiddle::Closure`, Lua's `ffi.cast`, .NET's
   `Marshal.GetFunctionPointerForDelegate`. Languages without runtime
   trampolines (C++, Rust, Zig) generate them at compile time, which is
   strictly easier.

3. **Pinning C-visible memory across calls.** GC'd languages need a
   keepalive collection for both function-pointer trampolines and for any
   bytes/strings ecewo stores by pointer. Non-GC'd languages get this for
   free.

4. **Threading rules.** ecewo runs on one libuv loop. Your host runtime
   must either own that loop directly (blocking `ecewo_listen` on the
   thread you call it from), or schedule it on a dedicated thread and
   marshal completions back. See
   [`17.ffi-bindings.md`](17.ffi-bindings.md#threading-model).

Everything above this section is a refinement of those four problems.
Get them right and the rest of the binding is rote: bind one function per
line, wrap requests/responses with thin ergonomic shells, add a plugin
module for each plugin you want to expose, ship a wheel.
